# Hypothesis Lab MVP 开发计划

## 1. 文档目标
本计划用于指导 `Hypothesis Lab / 策略实验室` 从产品定义阶段进入工程实现阶段。

计划的唯一核心目标不是“尽快堆功能”，而是以最小复杂度、最高稳定性，跑通并验证下面这条闭环：

`交易想法 -> 策略假设卡 -> 一键回测 -> 结果理解 -> 结论沉淀 -> 交易手册`

只要这条闭环没有稳定成立，任何额外功能都不应优先于核心链路。

## 2. 开发总原则
1. 闭环优先：所有开发任务都必须直接服务于“10 分钟内完成一次策略验证闭环”。
2. 模块化优先：MVP 阶段采用“模块化单体”架构，不拆微服务，先降低协作复杂度。
3. 领域模型优先：先定义清楚核心对象，再做页面和交互，避免前后端各自演化。
4. 正确性优先于丰富度：回测结果必须稳定、可复现、可解释，优先级高于页面炫酷和功能铺量。
5. 标准契约优先：模块之间只通过明确的数据结构和接口协作，避免隐式耦合。
6. Out of Scope 冻结：不做实盘、不做社区、不做 AI 预测、不做复杂量化平台能力。
7. 小步快跑：每个阶段必须有可演示、可验收、可回归的阶段成果，不做“大而全”的长周期开发。

## 3. 北极星与完成定义

### 3.1 北极星指标
`30 天内完成的策略假设闭环次数`

这里的“闭环”必须满足：

`创建策略 -> 发起回测 -> 查看结果 -> 写下结论`

### 3.2 MVP 完成定义
满足以下条件，才算 MVP 真正成立：

1. 用户可以创建并保存一张策略假设卡。
2. 用户可以通过模板化规则定义基础交易逻辑。
3. 用户可以发起一次真实回测并获得稳定结果。
4. 用户可以看懂核心指标、资金曲线和交易明细。
5. 用户可以保存结论，并决定是否纳入交易手册。
6. 所有关键数据在刷新后仍可恢复。
7. 同一输入重复回测时，结果保持一致。

### 3.3 体验目标
1. 首次从新建策略到拿到结果，目标不超过 10 分钟。
2. 策略创建过程应在 3 分钟内可完成基础输入。
3. 用户应在 30 秒内看懂结果好坏与风险水平。
4. 用户应在 1 分钟内完成一条基础结论。

## 4. 推荐总体架构

### 4.1 架构选择
MVP 阶段推荐采用：

`前端应用 + 模块化后端 + 可插拔回测引擎 + 单一主数据库`

不建议一开始拆微服务，原因如下：

1. 当前产品核心难点在于闭环验证，不在于分布式扩展。
2. 早期拆服务会放大接口治理、部署、排障、数据一致性的成本。
3. 模块化单体已经足以支撑清晰分工，同时保留后续拆分空间。

### 4.2 逻辑分层
| 层级 | 核心职责 | 边界说明 |
| --- | --- | --- |
| 表现层 | 首页、策略编辑页、结果页、交易手册页、交互反馈 | 不承载业务规则，不直接拼接回测算法 |
| 应用编排层 | 创建策略、触发回测、保存结论、加入手册、查询历史 | 负责流程，不负责底层计算 |
| 领域层 | 策略卡、规则模板、回测运行、结果快照、结论、手册条目 | 负责业务语义，是系统稳定性的核心 |
| 基础设施层 | 数据库、市场数据接入、指标计算、日志、配置、任务执行 | 只提供能力，不承载业务判断 |
| 运维质量层 | 测试、监控、错误追踪、备份、部署流水线 | 为稳定开发和上线提供保障 |

### 4.3 推荐技术路线
以下是兼顾开发效率与量化场景适配度的建议方案，可替换，但不建议在 MVP 阶段频繁摇摆：

| 领域 | 推荐方案 | 原因 |
| --- | --- | --- |
| 前端 | `Next.js / React + TypeScript` | 页面开发快，组件化清晰，适合快速迭代 |
| 后端 API | `FastAPI` | 适合 Python 生态，接口定义清晰，便于与回测引擎协同 |
| 回测引擎 | Python 独立领域模块 | 技术指标、数据处理、回测逻辑更贴近 Python 生态 |
| 数据库 | `PostgreSQL` | 结构化数据稳定，支持 JSON 扩展，适合 MVP 成长 |
| 图表 | `ECharts` 或同类图表库 | 足够支撑资金曲线、回撤曲线与基础交互 |
| 测试 | `Pytest + Playwright` | 同时覆盖算法正确性与关键闭环流程 |
| 部署 | 单体部署或前后端双部署 | 先保证稳定和可运维，不追求复杂基础设施 |

### 4.4 关键架构决策
1. 后端先做“模块化单体”，不做微服务。
2. 回测引擎从第一天开始独立为可替换模块，但先和主后端同仓协作。
3. 账户体系在 MVP 可采用单用户模式，但所有核心表都保留 `user_id` 字段。
4. 回测执行先支持同步接口或轻量异步封装，但必须预留 `run_id` 和状态机。
5. 结果摘要先做模板化解释，不做自由生成式 AI。

## 5. 核心领域模型
核心模型必须先冻结，再允许页面和接口开发。

| 对象 | 作用 | 关键原则 |
| --- | --- | --- |
| `StrategyCard` | 用户正在编辑和管理的策略假设卡 | 可编辑，可复制，是实验入口 |
| `StrategySnapshot` | 发起回测时固化的策略输入快照 | 一经生成不可修改，确保结果可追溯 |
| `BacktestRun` | 一次回测运行记录 | 有唯一 `run_id`、状态、耗时、错误信息 |
| `BacktestResult` | 回测输出结果 | 指标、曲线、交易明细、摘要分离存储 |
| `Conclusion` | 用户对一次实验的判断 | 必须与具体策略或回测结果关联 |
| `HandbookEntry` | 被沉淀进交易手册的策略条目 | 不直接修改回测结果，只引用已有结论与摘要 |
| `MarketDataset` | 标准化历史行情数据集 | 数据口径统一，版本清晰，可回归验证 |
| `RuleTemplate` | 模板化交易规则定义 | 由统一 schema 驱动，前后端共享含义 |

### 5.1 必须坚持的建模规则
1. 每次回测都必须绑定一个不可变的 `StrategySnapshot`。
2. `BacktestResult` 只允许由引擎生成，不允许页面层直接改写。
3. 结论与手册条目必须能追溯到对应策略与回测结果。
4. 策略卡可以不断编辑，但历史回测记录必须保留。
5. 指标定义必须统一，禁止前端和后端各算一套。

## 6. 模块拆分与职责边界

### 6.1 业务模块
| 模块 | 主责建议 | 核心职责 | 不负责什么 | 阶段优先级 |
| --- | --- | --- | --- | --- |
| 策略假设模块 | 前端 + 后端领域 | 新建、保存、编辑、复制策略卡 | 不执行回测，不生成结果摘要 | P0 |
| 规则模板模块 | 后端领域 | 定义规则模板、参数 schema、校验规则、标准化输出 | 不负责页面展示细节 | P0 |
| 回测执行模块 | 后端 / 算法 | 根据策略快照和行情数据执行回测 | 不负责用户文案和手册逻辑 | P0 |
| 结果分析模块 | 后端 | 生成核心指标、资金曲线、回撤曲线、交易明细、模板摘要 | 不负责策略编辑 | P0 |
| 结论沉淀模块 | 前后端协同 | 结论填写、保存、编辑、展示 | 不负责重新运行回测 | P0 |
| 交易手册模块 | 前后端协同 | 纳入手册、移出手册、展示已验证策略与结论 | 不负责回测算法 | P0 |
| 实验记录模块 | 前后端协同 | 策略列表、回测历史、最近实验记录 | 不负责复杂研究分析 | P1 |

### 6.2 底座模块
| 模块 | 主责建议 | 核心职责 | 不负责什么 | 阶段优先级 |
| --- | --- | --- | --- | --- |
| 市场数据模块 | 后端 / 数据 | 行情抓取、清洗、标准化、缓存、数据版本管理 | 不负责业务判断 | P0 |
| 持久化模块 | 后端 | 数据表设计、仓储接口、迁移脚本、备份恢复 | 不负责页面流程 | P0 |
| 账户与权限模块 | 后端 | 用户范围隔离、登录能力预留 | 不抢占闭环优先级 | P1 |
| 平台基础模块 | 全栈 | 配置管理、日志、错误追踪、测试基建、CI/CD | 不写业务逻辑 | P0 |

### 6.3 模块边界约束
1. 页面层只能消费后端已经定义好的领域契约，不自行推导业务规则。
2. 规则模板模块输出的必须是标准化策略配置，而不是 UI 私有结构。
3. 回测引擎只吃标准化配置和标准化数据，不依赖页面状态。
4. 结果分析模块和回测执行模块逻辑分离，避免“算指标”和“跑策略”耦合。
5. 交易手册模块只引用已存在的结论和结果，不重新派生事实数据。

## 7. 建议仓库结构
以下结构的目标不是“看起来高级”，而是从一开始就把职责分开：

```text
/apps
  /web                  前端应用
/services
  /api                  主后端 API 与应用编排
/packages
  /domain-contracts     领域模型与接口契约
  /backtest-engine      回测引擎
  /rule-templates       规则模板与校验
  /shared-utils         通用工具
/tests
  /integration
  /e2e
/docs
  产品文档、接口文档、测试样例、验收清单
```

如果当前团队规模很小，也可以暂时合并目录，但“契约、引擎、应用编排、前端”四类职责仍应逻辑分离。

## 8. 分阶段开发计划

### 阶段 0：架构定稿与工程底座
**目标**：在真正写业务功能前，把后续不会轻易变化的骨架先立住。

**核心产出**
1. 冻结 MVP P0 范围与 Out of Scope。
2. 冻结核心领域模型和数据库草案。
3. 定义规则模板 schema v1。
4. 建立项目目录、环境变量规范、日志规范、错误码规范。
5. 建立最小 CI 流水线与测试基线。
6. 接入 BTC/ETH 的 4H、1D 行情样本数据。
7. 用 Mock 数据打通前端页面骨架。

**阶段退出标准**
1. 前后端对 `StrategyCard`、`BacktestRun`、`BacktestResult` 的字段定义达成一致。
2. 本地开发环境可一键启动。
3. 至少有一份固定测试数据集可用于后续回归。

### 阶段 1：策略输入端落地
**目标**：让用户能快速、稳定地写出一张“可回测”的策略假设卡。

**核心产出**
1. 首页基础框架与“新建策略假设”入口。
2. 策略假设编辑页。
3. 策略名称、标的、周期、回测区间、手续费等基础字段。
4. 模板化入场、出场、止损规则表单。
5. 表单校验、保存、再次加载与编辑。
6. 标准化策略配置生成逻辑。

**阶段退出标准**
1. 用户可以在 3 分钟内完成一张基础策略卡。
2. 保存后的策略可以再次编辑。
3. 前端输出的数据结构可以直接供回测模块消费，无需额外手工转换。

### 阶段 2：回测核心能力落地
**目标**：让系统对标准化策略配置执行真实、稳定、可重复的历史回测。

**核心产出**
1. 标准化行情数据读取接口。
2. 基础指标计算能力。
3. 单标的、单策略、单周期、只做多的回测逻辑。
4. 初始资金、手续费、全仓进出等 MVP 规则实现。
5. 回测运行状态管理与错误处理。
6. `StrategySnapshot -> BacktestRun -> BacktestResult` 的完整链路。

**阶段退出标准**
1. 同一输入重复运行得到一致结果。
2. 回测失败时有明确错误信息。
3. BTC/ETH、1D/4H 的预设范围可稳定出结果。

### 阶段 3：结果阅读与认知闭环落地
**目标**：让用户不仅看到数字，还能完成判断和沉淀。

**核心产出**
1. 回测结果页。
2. 核心指标卡片。
3. 资金曲线和回撤曲线。
4. 最近交易明细。
5. 模板化结果摘要。
6. 结论填写区。
7. “加入交易手册”功能。

**阶段退出标准**
1. 用户可以在 30 秒内判断策略是否值得继续研究。
2. 用户可以保存结论并再次查看。
3. 用户可以把策略纳入交易手册，完整闭环首次成立。

### 阶段 4：可持续实验能力补齐
**目标**：让用户不再每次从零开始，而是能持续实验和复盘。

**核心产出**
1. 策略卡列表页。
2. 编辑、删除、复制策略卡。
3. 最近回测记录与历史记录。
4. 首页“最近实验”与“最近结果”区域。
5. 空状态提示与示例策略。

**阶段退出标准**
1. 用户可以围绕已有策略持续迭代。
2. 用户可以查看最近几次回测结果并继续优化。
3. “复制再调整”流程顺畅可用。

### 阶段 5：稳定性、上线与试运行
**目标**：让产品从“能演示”进入“可持续使用”。

**核心产出**
1. 性能优化与慢查询排查。
2. 日志、监控、错误追踪与告警。
3. 数据备份与恢复方案。
4. 核心链路 E2E 自动化测试。
5. 种子数据、默认策略、空页面引导。
6. 试运行环境部署与验收清单。

**阶段退出标准**
1. 关键闭环在试运行环境中稳定可用。
2. 回测错误可追踪，数据异常可定位。
3. MVP 达到可供真实连续使用的水平。

### 阶段 6：P1 增强能力
这一阶段不应抢占 P0 闭环优先级，建议在 MVP 稳定后进入：

1. 参数复制与优化加强。
2. 回测历史版本管理。
3. 参数对比视图。
4. 标签分类。
5. 首次使用引导与示例策略增强。

## 9. 模块依赖顺序与并行开发建议

### 9.1 必须先完成的依赖链
1. `规则模板 schema` 先于 `策略编辑表单`。
2. `标准化行情数据` 先于 `回测引擎`。
3. `回测结果字段定义` 先于 `结果页与摘要模块`。
4. `结论与手册的数据模型` 先于 `闭环验收`。

### 9.2 可并行推进的工作
1. 前端页面骨架可以基于 Mock 数据先做。
2. 市场数据模块和回测引擎可以并行开发。
3. 结果页 UI 可以在引擎完成前先接假数据联调。
4. 测试样例库可以与回测引擎同步建设。

### 9.3 最推荐的协作分工
| 工作流 | 主要负责人 | 交付内容 |
| --- | --- | --- |
| 产品与架构 | 产品负责人 / 架构负责人 | 边界控制、领域模型、阶段验收、优先级管理 |
| 前端体验 | 前端负责人 | 页面、交互、表单、状态反馈、结果可读性 |
| 后端编排 | 后端负责人 | API、应用流程、持久化、权限、日志 |
| 回测与数据 | 算法 / 后端负责人 | 行情数据、规则执行、指标计算、回测正确性 |
| 测试验收 | 全员共担 | 回归样例、E2E、阶段验收、试运行反馈 |

如果当前是单人开发，上述职责也必须在思维上拆开，按模块推进，不要“页面写一点、算法写一点、数据库改一点”地来回跳。

## 10. 数据与接口治理要求
这是保证“模块化不失控”的关键部分。

### 10.1 必须冻结的契约
1. `StrategyCard` 的表单字段与标准化输出结构。
2. `BacktestRequest` 的输入字段。
3. `BacktestResult` 的指标定义与返回结构。
4. `Conclusion` 与 `HandbookEntry` 的关联关系。

### 10.2 指标口径必须统一
以下指标必须由后端统一计算，并以同一口径返回前端：

1. 总收益率
2. 最大回撤
3. 胜率
4. 盈亏比
5. 交易次数
6. 平均持仓时长

### 10.3 结果存储原则
1. 每次回测都必须保存输入快照和结果快照。
2. 页面上的展示文案必须来自已保存的结果数据，而不是临时推导。
3. 模板化摘要必须可以被重新生成，但原始结果快照必须保留。

## 11. 测试与质量保障计划

### 11.1 测试分层
| 测试层级 | 重点目标 | 代表内容 |
| --- | --- | --- |
| 单元测试 | 保证规则校验、指标计算、回测逻辑稳定 | 规则模板校验、收益率计算、回撤计算 |
| 集成测试 | 保证模块间契约正确 | 策略保存后发起回测、结果入库、结论绑定 |
| E2E 测试 | 保证核心闭环可真实完成 | 新建策略 -> 回测 -> 看结果 -> 写结论 -> 加入手册 |
| 回归测试 | 保证同一策略结果不漂移 | 固定历史数据集 + 固定策略样例 |

### 11.2 必须建立的回归样例
建议至少维护以下 3 组标准样例策略：

1. 均线穿越策略。
2. RSI 阈值策略。
3. 突破策略。

这些样例不是产品功能，而是工程稳定性的“标准尺子”。

### 11.3 质量门禁
任何进入主分支的功能至少满足：

1. 契约文档已更新。
2. 单元测试与集成测试通过。
3. 关键错误状态有明确提示。
4. 不引入超出本阶段范围的功能扩张。

## 12. 发布节奏建议
为了让开发有序平稳推进，建议采用短周期节奏：

| 周期 | 目标 |
| --- | --- |
| 每周开始 | 确认本周只做一个阶段目标，不临时塞入 P1 |
| 每周中段 | 做一次集成演示，提前暴露接口或数据问题 |
| 每周结束 | 按阶段退出标准做一次验收，而不是只看代码是否写完 |

## 13. 风险清单与控制策略
| 风险 | 表现方式 | 控制策略 |
| --- | --- | --- |
| 范围失控 | 很快扩展到内容、社区、AI、复杂量化 | 严格冻结 Out of Scope，只围绕闭环做决策 |
| 数据口径不一致 | 前后端看到的收益、回撤、胜率不一致 | 指标统一由后端生成，前端只展示 |
| 回测结果不稳定 | 同一策略重复回测得到不同结果 | 固化输入快照、固定数据版本、建立回归样例 |
| 过早优化架构 | 花大量时间在服务拆分、基础设施 | 坚持模块化单体，先跑通真实闭环 |
| 只做页面不做正确性 | 演示好看但无法真实使用 | 把“回测可复现”列为硬验收项 |
| 总结与结果脱节 | 摘要看起来合理但和数据矛盾 | 摘要模板严格绑定指标阈值规则 |

## 14. 工期建议
以下工期是以“目标清晰、需求稳定”为前提的粗估：

| 团队配置 | 预估周期 |
| --- | --- |
| 单人开发 | 10-12 周 |
| 2 人协作 | 6-8 周 |
| 3 人小团队 | 5-7 周 |

工期不应以“页面数量”估算，而应以“闭环是否真实成立”估算。

## 15. 立即开工的第一批任务
为了让计划快速进入执行，建议按下面顺序启动：

1. 冻结 P0 范围和 Out of Scope。
2. 输出核心领域模型和数据库初稿。
3. 确定规则模板 v1 与标准化策略配置结构。
4. 确定市场数据来源、格式与样本导入流程。
5. 建立项目目录、环境、日志、测试基线。
6. 用 Mock 数据完成首页、策略编辑页、结果页骨架。
7. 开始实现真实回测引擎与结果快照结构。
8. 打通第一次完整闭环后，再补列表、复制、历史记录。

## 16. 最终结论
这个项目的开发计划不应围绕“页面数”展开，而应围绕“认知闭环是否被稳定实现”展开。

从顶层架构视角看，最正确的路线不是一开始做成大平台，而是：

1. 用模块化单体保证结构清晰。
2. 用标准契约保证模块解耦。
3. 用阶段化交付保证推进有序。
4. 用回归样例保证结果可信。
5. 用交易手册作为长期价值出口。

只要你坚持这条路线，`Hypothesis Lab` 的 MVP 会更像一个真正可增长的产品雏形，而不是一组零散功能的拼接。
