# 技术方案与技术约束

## 1. 文档目的
本文件用于在正式编码前冻结 MVP 阶段的核心技术路线与硬性约束，避免开发过程中频繁摇摆。

## 2. 当前冻结技术方案
| 领域 | 冻结方案 | 备注 |
| --- | --- | --- |
| 前端框架 | `Next.js + React + TypeScript` | 页面与交互主载体 |
| 前端状态策略 | `React Hook Form + Zod + TanStack Query + 组件局部状态` | MVP 阶段不引入全局状态库 |
| 前端包管理 | `pnpm` | 适合 monorepo |
| 后端 API | `FastAPI` | 负责应用编排与接口输出 |
| Python 包管理 | `uv` | 用于虚拟环境、依赖管理和运行命令 |
| 数据迁移 | `Alembic` | 配合 `PostgreSQL` 使用 |
| 回测引擎 | Python 独立领域模块 | 与主后端同仓协作 |
| 数据库 | `PostgreSQL` | 主业务数据存储 |
| 图表 | `ECharts` 或同类库 | 用于资金曲线和回撤曲线 |
| 自动化测试 | `Pytest + Playwright` | 覆盖算法正确性与核心闭环 |

## 3. MVP 阶段硬约束
1. 架构采用模块化单体，不拆微服务。
2. 回测引擎必须独立为可替换模块，但先与主后端同仓协作。
3. 所有核心指标由后端统一计算，前端只做展示。
4. 结果摘要先做模板化解释，不做生成式 AI。
5. 只支持加密市场、`BTCUSDT`、`ETHUSDT`，周期只支持 `4H` 与 `1D`。
6. 只支持单标的、单策略、单周期、只做多、全仓进出。
7. MVP 运行在单用户模式，但所有核心表保留 `user_id`。
8. 回测接口从第一天起采用“轻量异步 + 轮询”契约，不要求先上消息队列。
9. 不做实盘、不接交易所、不做社区、不做课程、不做复杂量化平台。

## 4. 工程约束
1. 所有跨模块结构先在 `docs/contracts/` 冻结，再进入实现。
2. 同一策略输入重复回测，结果必须一致。
3. 每次回测都必须保留输入快照和结果快照。
4. 任何新增能力不得破坏 MVP 核心闭环优先级。
5. 阶段 0 未完成前，不允许前后端各自定义私有契约并并行开写。

## 5. 本地开发约定
1. 顶层仓库继续作为单一协作仓库使用。
2. 前端依赖通过 `pnpm` 管理，后端依赖通过 `uv` 管理。
3. 数据库迁移统一通过 `Alembic` 管理，不允许手写“只在本地执行一次”的 SQL 作为长期方案。
4. 本地启动命令和环境变量清单应统一记录在 `docs/ops/deployment-and-environment.md` 与 `.env.example`。

## 6. 当前阶段的明确取舍
1. 前端先不引入 Zustand/Redux 类全局状态库。
2. 回测异步能力先在进程内实现，后续如有需要再抽离任务执行器。
3. `RuleTemplate` 优先作为代码注册表存在，不要求第一天就建数据库表。
4. 先保证主链路可跑通，再补 CI 丰富度和部署复杂度。

## 7. 更新规则
若未来出现技术路线调整，必须同时更新：
1. 本文档
2. `docs/architecture/system-architecture-and-module-boundaries.md`
3. 对应 ADR
