# 技术方案与技术约束

## 1. 文档目的
本文件用于在正式编码前冻结 MVP 阶段的核心技术路线与硬性约束，避免开发过程中频繁摇摆。

## 2. 当前推荐技术方案

| 领域 | 当前建议 |
| --- | --- |
| 前端 | `Next.js + React + TypeScript` |
| 后端 API | `FastAPI` |
| 回测引擎 | Python 独立领域模块 |
| 数据库 | `PostgreSQL` |
| 图表 | `ECharts` 或同类图表库 |
| 自动化测试 | `Pytest + Playwright` |

## 3. MVP 阶段硬约束
1. 架构采用模块化单体，不拆微服务。
2. 回测引擎必须独立为可替换模块，但先与主后端同仓协作。
3. 所有核心指标由后端统一计算，前端只做展示。
4. 结果摘要先做模板化解释，不做生成式 AI。
5. 只支持加密市场、BTC/ETH、`4H` 与 `1D`。
6. 只支持单标的、单策略、单周期、只做多、全仓进出。
7. 不做实盘、不接交易所、不做社区、不做课程、不做复杂量化平台。

## 4. 工程约束
1. 所有跨模块结构先在 `contracts/` 冻结，再进入实现。
2. 同一策略输入重复回测，结果必须一致。
3. 每次回测都必须保留输入快照和结果快照。
4. 任何新增能力不得破坏 MVP 核心闭环优先级。

## 5. 暂未冻结但需要尽快确认
1. 前端状态管理方案。
2. Python 包管理方案。
3. 数据迁移工具方案。
4. 本地开发启动方式。

## 6. 更新规则
若未来出现技术路线调整，必须同时更新：

1. 本文档
2. `docs/architecture/system-architecture-and-module-boundaries.md`
3. 对应 ADR
