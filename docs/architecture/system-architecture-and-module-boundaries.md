# 系统架构与模块边界

## 1. 目标
本文件用于明确系统分层、模块职责、输入输出边界，以及哪些模块可以独立演进，哪些模块必须共享统一契约。

## 2. 总体架构

```text
Web UI
  -> API / Application
    -> Domain
      -> Rule Templates
      -> Backtest Engine
      -> Persistence / Market Data
```

## 3. 分层定义

| 层级 | 职责 | 禁止事项 |
| --- | --- | --- |
| 表现层 | 页面、交互、状态反馈 | 不直接实现业务规则和回测逻辑 |
| 应用编排层 | 串联创建策略、发起回测、保存结论、加入手册 | 不直接写底层计算 |
| 领域层 | 核心业务语义与规则约束 | 不处理 UI 细节 |
| 基础设施层 | 数据库、行情数据、日志、配置 | 不承载业务判断 |

## 4. 业务模块边界

| 模块 | 输入 | 输出 | 说明 |
| --- | --- | --- | --- |
| 策略假设模块 | 页面表单输入 | 标准化策略配置 | 只负责表达和保存策略 |
| 规则模板模块 | 模板类型与参数 | 标准规则结构 | 不负责页面展示 |
| 回测执行模块 | 策略快照、行情数据 | 原始交易结果 | 不负责用户文案 |
| 结果分析模块 | 原始交易结果 | 指标、曲线、摘要 | 不负责策略编辑 |
| 结论沉淀模块 | 用户输入结论 | 已保存结论 | 不负责回测执行 |
| 交易手册模块 | 已有结果与结论 | 手册条目 | 不重新计算事实数据 |

## 5. 模块间硬边界
1. 页面层只能消费后端公开契约。
2. 回测引擎不能依赖前端结构。
3. 结果分析与回测执行逻辑分离。
4. 手册模块只引用结果与结论，不派生新的结果事实。
5. 任何跨模块字段新增都必须先更新 `contracts/`。

## 6. 当前最关键的接口边界
1. `StrategyCard -> StrategySnapshot`
2. `StrategySnapshot -> BacktestRun`
3. `BacktestRun -> BacktestResult`
4. `BacktestResult -> Conclusion`
5. `Conclusion -> HandbookEntry`
